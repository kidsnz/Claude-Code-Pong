# cc-pong.asm ステップ計画

APONG（`docs_pong/AtariAge/APongJuly02.bin`）のビジュアル再現を目的とした制作計画。

## 設計の前提

- **完成条件**: 各ステップで `dasm` がエラーなく通り、Stella で指定要素が見える
- **テスト**: ユーザーが Stella で確認 → OK で Git コミット → 次へ
- **APONG式カーネル**: PF0=パドル, Missile0=ボール, Ball=ネット, GRP0/1=スコア
- **基本原則**: 静的描画 → 動き → 物理 → ゲームロジックの順
- **コントローラ**: ジョイスティック（SWCHA）

---

## Phase A — 骨格

### Step 1: スケルトン
- VSYNC（3行）+ VBLANK（37行）+ Kernel（192行）+ Overscan（30行）= 262行 NTSC
- TIM64T でフレーム終端を待つ（ハングしない正確なフレーム）
- 全黒画面
- **完成条件**: Stella で真っ黒な安定画面が出る（フリッカー無し）

## Phase B — 静的ビジュアル

### Step 2: センターネット
- BALL レジスタ + VDELBL でダッシュパターン
- 画面中央に縦の点線（最も簡単な1要素から始める＝カーネル時間検証）
- **完成条件**: 中央に縦ダッシュラインが見える

### Step 3: 静的スコア "0-0"
- **2ゾーンカーネル確立**: 上部=スコアゾーン、下部=フィールド
- GRP0/GRP1 + DigitPtr 間接アドレッシング、7セグメント数字テーブル
- **完成条件**: 上部に "0  0" の大きな数字が出る（ネットは継続表示）

### Step 4: 静的パドル
- フィールドゾーンに PF0 で左右パドル描画
- 最初は簡素な実装（line counter＋compare、DCP最適化は後）
- 左右ともに固定 Y で表示
- **完成条件**: 左右両端に白い小さな矩形パドル

### Step 5: 静的ボール
- Missile0、ENAM0、RESM0 + HMM0 でフィールド中央に配置
- **完成条件**: フィールド中央に小さい白い正方形

## Phase C — 入力

### Step 6: パドル移動
- SWCHA でジョイスティック上下読み取り（P0_Y, P1_Y 更新）
- 移動範囲: スコア下端 〜 画面最下部にクランプ
- **完成条件**: 1P ジョイスティックで左パドル、2P で右パドルが動く

## Phase D — 物理

### Step 7: ボール自動移動＋壁反射
- BallX/BallY と BallVX/BallVY
- フィールド上下端で Y 速度反転
- 左右端は今は素通り（次のステップで処理）
- **完成条件**: ボールが画面内を斜めに動き、上下で跳ね返る

### Step 8: パドル衝突反射
- ボール X がパドル X 位置に到達 + ボール Y がパドル Y 範囲内 → X 速度反転
- TIA 衝突レジスタ（CXM0FB）使用も検討
- **完成条件**: パドルにボールが当たると跳ね返る

## Phase E — ゲーム

### Step 9: 得点・サーブ
- ボールがパドルの外側に出る → 相手スコア +1、ボール中央リセット
- サーブ方向は失点側に向ける（交互）
- **完成条件**: 失点でスコアが増え、ボールが中央から再開

### Step 10: ゲーム終了・リスタート
- 一定スコア（例: 15点）で勝者を点滅 or 静止
- RESET スイッチで初期化
- **完成条件**: 最後まで遊べる、リセットで初期状態へ

---

## Phase F — 仕上げ（任意）

### Step 11: サウンド
- 壁ヒット / パドルヒット / 得点 の3種

### Step 12: モード切替
- SELECT スイッチで 1P（CPU 相手）⇔ 2P

---

## 設計判断の根拠

| 判断 | 理由 |
|------|------|
| **ネットを2番目（パドルより先）** | レジスタ1つだけの最小要素。基本カーネル時間が正しいか先に検証 |
| **スコアをパドルより先** | 2ゾーンカーネル構造を早期に確立。後でカーネル構造を変えるのは破綻リスク |
| **静的描画を全て先に** | 動きのバグは静的描画のバグより遥かに追いにくい |
| **パドル移動 → ボール移動** | パドルは入力→1変数更新のみ。ボールは速度・反射・端判定で複雑 |
| **DCP 最適化は後回し** | まず動くものを作り、サイクル詰めは必要になったらやる |
